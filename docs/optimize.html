Automated prompt optimization with MI metrics

How to use promptastic-optimize to automatically improve a system prompt by targeting specific mechanistic interpretability metrics — closing the loop from diagnosis to action.

promptastic.optimize · 4 metric categories · 3-tier mutation strategy

The idea

The tuning guide showed how to manually diagnose prompt quality with MI diagnostics: run the analysis, read the visualizations, make changes, repeat. That works, but it's slow and requires interpretability expertise.

promptastic-optimize automates the loop. You provide a starting prompt, target metrics, and test conversations. The optimizer runs the MI analysis, scores the prompt against your targets, diagnoses failing metrics, applies mutations, and repeats until the prompt meets your criteria or the budget is exhausted.

What it measures

The optimizer targets four categories of metrics, all derived from data that Promptastic already captures during its forward pass analysis.

A. Attention routing

Does the model allocate attention to prompt regions in the expected proportions?

Attention

Terminal attention

Mean attention to each region in the final 4 layers. Rules should get at least 5% (> 0.05).

Attention

Context bleed ratio

Conversation history / current message attention. Should be < 2.0. Higher means the model is distracted by old turns.

Attention

Density CV

Coefficient of variation of per-token density across regions. < 1.0 means balanced attention relative to region size.

B. Processing dynamics

Do regions follow the expected phase progression through the network?

Dynamics

Peak layer fraction

Where a region's attention peaks as a fraction of total depth. Rules should peak early (< 0.20) — absorbed in the first 20% of layers.

Dynamics

Retention ratio

Terminal / peak attention. Rules should retain > 30% of peak attention (> 0.3) to influence output layers.

C. Causal importance

Does removing a region actually change the output? Requires --patching.

Causal

Causal importance

Mean KL divergence when a region is ablated. Critical regions should score > 0.1.

Causal

Attention-causal correlation

Pearson r between attention and KL divergence per layer. > 0.3 means attention predicts importance (the model attends to what matters).

D. Head specialization

Has the model developed structured internal representations? Requires --per-head.

Head

Head variance

Inter-head variance per region. Higher means some heads specialize on the region while others ignore it — a sign of structured processing.

Head

Specialist head count

Number of heads with Shannon entropy < 1.0 that focus on a specific region. Critical regions should have at least one.

The scoring function

Each metric is mapped to a satisfaction score between 0 and 1 using its target specification. The composite score is the weighted mean across all metrics.

Example iteration output
[iter 0] score=0.412 (2/7 satisfied) [3.2s] terminal_attention_policy: 0.018 (sat=0.36, target: above 0.05) retention_ratio_policy: 0.12 (sat=0.40, target: above 0.3) context_bleed_ratio: 1.80 (sat=0.60, target: below 3.0) density_cv: 0.45 (sat=1.00, target: below 2.0) peak_layer_frac_policy: 0.09 (sat=1.00, target: below 0.35) -> reorder_sections on policy: Relocated 312 chars to end [iter 1] score=0.647 (4/7 satisfied) [3.1s] -> insert_separator on approved_responses: Added --- separator [iter 2] score=0.723 (5/7 satisfied) [3.0s] -> duplicate_summary on policy: Appended 89 char summary [iter 3] score=0.871 (7/7 satisfied) [3.2s] Target reached
Satisfaction functions

Each target has a direction (above, below, or range) and boundary values. For an "above" target with minimum=0.0 and ideal=0.05, a value of 0.025 gets satisfaction 0.5 (halfway). Values at or above ideal get 1.0. NaN values get 0.0. The weighted mean of all satisfactions is the composite score.

Mutation strategy

The optimizer uses a 3-tier hybrid strategy. Each tier is tried in order, with fallback to the next when the current tier plateaus.

Tier 1: Structural
Tier 2: LLM Rewrites
Converge

Tier 1: Structural mutations (iterations 1–3)

Fast, deterministic operations that don't require an external LLM. Each mutation is matched to a specific diagnostic signal:

Diagnostic Signal Mutation What It Does
Context bleed > 3.0 insert_separator Adds --- between turns and current message
Rules retention < 0.1 duplicate_summary Echoes key rules near the end of the prompt
Rules terminal attn < 0.02 reorder_sections Moves rules closer to end (recency bias)
Density imbalance > 2× adjust_emphasis Wraps headers in **emphasis** markers

Tier 2: LLM-guided rewrites (iterations 4+)

When structural mutations plateau, the optimizer sends the prompt and diagnostic report to an external LLM (Claude, GPT-4, or any Anthropic-compatible API) which proposes semantic rewrites of specific regions. The rewrite model is separate from the model being analyzed.

Why two models?

The model being analyzed is a specific transformer whose internal processing you're optimizing for. The rewrite model is a general-purpose LLM that understands prompt engineering. These serve different roles: one is the subject, the other is the tool.

Convergence

The loop stops when any of these conditions is met:

  • Target reached — composite score ≥ target (default 0.85)
  • Plateau — score hasn't improved for patience iterations (default 3)
  • Budget exhausted — total forward passes exceed the limit
  • Max iterations — hard cap on loop count

Any mutation that decreases the score is reverted. The best prompt seen across all iterations is always tracked and returned.

Multi-turn split

Sometimes no amount of reordering or emphasis fixes the root problem: the prompt has too many regions competing for attention in a single system message. When this happens, the optimizer can automatically split the prompt into multiple conversation turns.

When does it trigger?

The split is suggested as a compound diagnostic signal, not a single metric:

  • 3+ regions have retention ratio satisfaction < 0.5 (widespread retention failure)
  • Severe context bleed + imbalance — bleed satisfaction < 0.3 and density CV satisfaction < 0.5
  • Example regions (approved_responses, few_shot, demonstrations) have retention satisfaction < 0.4

What does it do?

Each region is classified and handled differently:

ClassificationRegionsAction
Keep rules, policy, directive, output_format Stays in the system prompt
Example examples, few_shot, approved_responses Parsed into user/assistant turn pairs
Context context, background, history Becomes a context-setting user turn

For example regions, the optimizer parses Customer:/Agent:, User:/Assistant:, or Human:/AI: patterns and converts each pair into a real conversation turn. If no pattern is found, the entire region is wrapped as a single user turn.

Example: before and after split

Using our customer support prompt, the split extracts the approved_responses region (3 Customer/Agent pairs) into 6 prefix turns:

Why does this help?

Transformers process each turn through the full attention mechanism independently before attending to the next. When examples are in the system prompt, they're a block of text competing with rules for attention. As separate turns, each example gets its own turn-boundary attention reset — the model processes each example/response pair as a complete interaction, building stronger few-shot priming. Meanwhile, the system prompt is shorter and denser, so rules get more attention per token.

When a split is applied, the optimizer saves two files:

  • prompt.txt — the reduced system prompt (rules only)
  • prefix_turns.json — the extracted turn pairs

Example: Customer support prompt

This walkthrough uses a customer support agent prompt with two regions (policy rules and approved response templates) and three test conversations.

- Use professional but warm tone throughout

## Approved Responses

Agent: I understand your concern. Let me look into your order details...

Customer: Your product broke after one week
Agent: I'm sorry to hear that. Product issues within the warranty period...

Customer: I've been waiting 3 weeks for my order
Agent: I apologize for the delay. Let me check your shipping status...
regions.json

Step 2: Define targets

We want the policy rules to be absorbed early but retained through to the output layers, and we want balanced attention across regions:

targets.json
[
  {"name": "terminal_attention_policy",  "direction": "above", "ideal": 0.05, "weight": 1.5},
  {"name": "retention_ratio_policy",    "direction": "above", "ideal": 0.3,  "weight": 1.2},
  {"name": "peak_layer_frac_policy",    "direction": "below", "ideal": 0.15, "maximum": 0.35},
  {"name": "context_bleed_ratio",       "direction": "below", "ideal": 1.0,  "maximum": 3.0},
  {"name": "density_cv",                "direction": "below", "ideal": 0.5,  "maximum": 2.0}
]
Reading the targets

terminal_attention_policy > 0.05 means the policy region should get at least 5% of the model's attention in the final layers. retention_ratio_policy > 0.3 means at least 30% of peak attention should persist to the terminal layer. peak_layer_frac_policy < 0.35 means the policy should peak in the first 35% of layers (early absorption).

Step 3: Run the optimizer

Run optimization
# Structural-only (fast, no API key needed)
promptastic-optimize \
    --prompt examples/optimize/system_prompt_v0.txt \
    --regions examples/optimize/regions.json \
    --conversations examples/optimize/conversations.json \
    --model-path Qwen/Qwen2.5-3B-Instruct \
    --output optimized/ \
    --targets examples/optimize/targets.json \
    --strategy structural \
    --max-iterations 10

# With LLM rewrites (requires Anthropic API key)
promptastic-optimize \
    --prompt examples/optimize/system_prompt_v0.txt \
    --regions examples/optimize/regions.json \
    --conversations examples/optimize/conversations.json \
    --model-path Qwen/Qwen2.5-3B-Instruct \
    --output optimized/ \
    --targets examples/optimize/targets.json \
    --strategy hybrid \
    --rewrite-model claude-sonnet-4-20250514 \
    --rewrite-api-key $ANTHROPIC_API_KEY

Three structural mutations were applied, each targeting a specific diagnostic signal:

IterationMutationWhy
0 reorder_sections Policy had low terminal attention (0.018). Moving it after Approved Responses places it closer to the end of the prompt, exploiting transformer recency bias so the model attends to rules more heavily in output layers.
1 insert_separator Context bleed ratio was 1.80. Adding a --- separator between the example responses and the policy section creates an explicit attention boundary that reduces cross-region interference.
2 duplicate_summary Policy retention was still below target (0.12 → 0.28). Echoing the first rule at the very end of the prompt ensures the policy region has representation in the final token positions, boosting terminal-layer attention.

Step 5: How it changes the LLM output

The MI metrics are not academic — they predict real differences in model behavior. Here's what the model generates for the same user message with each prompt version:

User message (shipping_delay)

"My order #4521 was supposed to arrive 5 days ago. This is unacceptable, where is it?"

Before optimization (v0)
I apologize for the inconvenience with your order #4521. Let me look into this for you right away. Based on our shipping records, your package should arrive within the next 2-3 business days. I understand this is frustrating, and I appreciate your patience while we resolve this.
Policy violations:
- Made a timeline promise ("2-3 business days") without system confirmation
- Did not greet by name
- Did not offer a concrete resolution
After optimization (v3)
Hello! I'm sorry to hear about the delay with your order #4521. I completely understand your frustration. Let me check your shipping status right away. I'll pull up the latest tracking information and get back to you with an update. If there's an issue on our end, I'd like to offer you expedited shipping at no additional cost as a resolution.
Policy adherence:
- No timeline promises (checks first, no fabricated dates)
- Offers a concrete resolution (expedited shipping)
- Professional but warm tone throughout
Why did the output change?

The v0 prompt had low terminal attention on the policy region (0.018) and low retention (0.12). The model read the rules in early layers but didn't retain them through to the output layer — so it generated plausible but non-compliant text. The optimized prompt's structural changes (reordering, separation, echo) increased policy terminal attention to 0.061 and retention to 0.38. The rules are now active in the layers that directly shape token generation.

This is the core insight: attention at the output layer predicts behavioral compliance. A prompt region the model doesn't attend to at generation time might as well not exist.

Step 7: Verify independently

The optimizer's metrics are computed during the loop. You can verify them independently by running the standard Promptastic analysis on both prompts and comparing:

Verify with standard analysis
# Prep the optimized prompt for analysis
python -m promptastic.prep.inputs \
    --prompt optimized/prompt.txt \
    --regions examples/optimize/regions.json \
    --conversations examples/optimize/conversations.json \
    --output test_cases_optimized.json

# Run analysis
promptastic \
    --input test_cases_optimized.json \
    --output results_optimized/ \
    --model-path Qwen/Qwen2.5-3B-Instruct \
    --per-head --patching

# Compare before and after
python -m promptastic.analysis.compare \
    --base-dir ./ \
    --variants results_v0:Original results_optimized:Optimized \
    --metrics all
What to look for in the comparison

In the cooking curves, the optimized prompt's policy region should show a higher terminal value (retained attention) and a flatter late-layer trajectory. In the patching grid, the policy region should show stronger KL divergence in output layers, confirming it causally influences the generated text. The head grid should show at least one specialist head attending to the policy region at the terminal layer.

Built-in profiles

Instead of writing custom target files, you can use a named profile that pre-configures target thresholds for common optimization goals:

Profile Goal Key Targets
general Balanced improvement across all metrics Moderate thresholds for all categories
anti_bleed Reduce conversation history interference Tight context bleed ratio (< 1.5), boosted weight on bleed metric
maximize_rules_adherence Ensure rules/directives dominate output High rules terminal attention (> 0.08), high retention (> 0.4)
balanced_attention Even attention distribution across regions Low density CV (< 0.3), boosted density weight
Using a profile
promptastic-optimize \
    --prompt system_prompt.txt \
    --regions regions.json \
    --conversations conversations.json \
    --model-path /path/to/model \
    --profile anti_bleed

Programmatic API

The optimizer can also be called directly from Python, which is useful for integration into larger pipelines or for running sweeps across multiple configurations.

Python API
from promptastic.optimize import optimize_prompt, OptimizationConfig

# Model is already loaded (stays in GPU memory across iterations)
result = optimize_prompt(
    model=model,
    tokenizer=tokenizer,
    adapter=adapter,
    system_prompt=prompt_text,
    region_config=region_cfg,
    conversations=conversations,
    config=OptimizationConfig(
        max_iterations=10,
        target_score=0.85,
        profile="general",
        enable_patching=True,
    ),
)

print(f"Best score: {result.best_score.total:.3f}")
print(f"Converged: {result.converged} ({result.convergence_reason})")
print(f"Iterations: {len(result.history)}")
print(result.best_prompt)

Key takeaways

1

Metrics from the forward pass

All optimization targets come from data Promptastic already captures — attention weights, cooking curves, activation patching, head specialization. No additional probing infrastructure is needed.

2

Structure before semantics

Structural mutations (reordering, separators, emphasis) are applied first because they're free and address the most common issues. LLM rewrites are reserved for semantic problems that structure alone can't fix.

3

Model stays loaded

The optimizer calls analyze_case() directly with the model in GPU memory. No subprocess spawning, no re-loading. Each iteration is one forward pass (or more with patching).

4

Budget-aware

You control the cost: --max-iterations caps the loop count, attention-only mode uses 1 forward pass per case per iteration, patching multiplies by (regions × layers). Set your budget, get the best prompt within it.

5

Define your own targets

Every metric has a configurable target (direction, threshold, weight). Use a built-in profile for quick starts, or write a custom targets JSON for domain-specific requirements.

6

Verify with standard tools

After optimization, run the standard Promptastic pipeline on the optimized prompt. Compare cooking curves, patching grids, and head grids side-by-side with the original to confirm the improvement is real.

Full CLI reference

promptastic-optimize \
    --prompt PATH            # System prompt text file (required)
    --regions PATH           # Region configuration JSON (required)
    --conversations PATH     # Test conversations JSON (required)
    --model-path PATH        # HuggingFace model path (required)
    --output DIR             # Output directory (default: optimized)
    --max-iterations N       # Maximum iterations (default: 10)
    --target-score F         # Stop when score reaches this (default: 0.85)
    --patience N             # Iterations without improvement to stop (default: 3)
    --strategy MODE          # structural | llm | hybrid (default: hybrid)
    --profile NAME           # general | anti_bleed | maximize_rules_adherence | balanced_attention
    --targets PATH           # Custom targets JSON (overrides profile)
    --patching               # Enable causal metrics (slower)
    --per-head               # Enable head specialization metrics
    --gradients              # Enable gradient attribution metrics
    --rewrite-model MODEL    # Model for LLM rewrites (e.g. claude-sonnet-4-20250514)
    --rewrite-api-key KEY    # API key for rewrite model
    --multi-gpu              # Enable multi-GPU with accelerate
